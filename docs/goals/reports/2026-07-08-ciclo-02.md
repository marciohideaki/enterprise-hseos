## 1. O que o ciclo entregou

### 1.1 Fonte canônica de hooks movida para `.enterprise/` (auto-referência eliminada)

O registry de hooks era o **único ponto do fluxo `.enterprise → .agents → adapters` que se auto-referenciava**: `.agents/hooks/registry.yaml` era simultaneamente input e output do compiler, e os handlers viviam apenas na árvore compilada.

Agora:
- **Fonte canônica:** `.enterprise/governance/hooks/{registry.yaml, handlers/}` (23 arquivos: 22 scripts + README) — hooks e handlers editam-se aqui, como as skills.
- **Compilado:** `agent-core compile` lê a fonte enterprise (precedência nova), regenera `.agents/hooks/registry.yaml` (com header "DO NOT EDIT" + proveniência `source:`) e **copia os handlers** para `.agents/hooks/handlers/` (CRLF-normalizados, mode preservado).
- **Integridade estendida:** o manifest ganhou `handlers[]` com `{file, source, sha256}` e o `agent-core verify` os checa como checa skills — **de 65 para 88 checks** (49 skills + 16 agents + 23 handlers). Handler editado sem recompilar agora é drift detectado.
- **Migração compatível:** fallback em cadeia (enterprise → `.agents` target → source root → hooks.json legado); projetos pré-migração compilam byte-idêntico (sem dir enterprise, o sync de handlers é no-op e o manifest mantém o formato anterior).

### 1.2 Hooks bloqueantes ganharam testes de comportamento

`test/test-blocking-hooks.js` (novo, no `npm test` via `test:hooks`) — **11 cenários herméticos** (repos git temporários; `CLAUDE_PROJECT_DIR` isolado para não sofrer interferência de runs históricos):

| Hook | Cenários |
|---|---|
| `swarm-gate` | execução sem model → ask; com sonnet → allow; com opus sem opt-in → ask; padrão dev-squad sem run → ask (gate); com flag de run ativo → allow (bypass) |
| `claude-md-guard` | CLAUDE.md via arg → deny; via payload stdin → deny; outro arquivo → allow silencioso |
| `ado-branch-guard` | push a trunk com `ado.enabled` → exit 2; push a feature → exit 0; flag off → no-op mesmo para trunk |

Antes deste ciclo, o enforcement bloqueante rodava **sem nenhum teste**; a promoção do swarm-gate canônico no ciclo 01 tinha sido validada só por smoke manual.

## 2. Aderência

- Elimina a violação estrutural apontada no relatório arquitetural do ciclo 01 (§1, "smell arquitetural") — o fluxo fonte→compilado agora é unidirecional em TODOS os subsistemas.
- P3 (integridade hash-pinned, ADR-0006) passa a cobrir a camada de enforcement em tempo real, não só skills/agents.
- Compatibilidade reversa preservada (prioridade 7 do goal): fallback em cadeia + manifest aditivo.
