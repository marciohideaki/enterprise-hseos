## 1. O que o ciclo entregou

### 1.1 Gate de segurança: de no-op silencioso a scan real (bug crítico)

Descoberta com prova empírica: **`gate_security` nunca escaneou nada** — `grep --include="*.{js,ts,...}"` usa brace-glob que o grep não expande, então o filtro não casava arquivo algum e o gate passava vácuo desde a origem. Correção:
- Scan via **`git grep` sobre arquivos rastreados** (gitignored nunca gateiam; CI escaneia exatamente o que é versionado), pathspecs por tipo + exclusões (`quality-gates.sh` se auto-excluí — contém os padrões como literais; `test/fixtures`).
- Padrões corrigidos (o `\x27` do original também não era ERE válido) e ampliados (`OPENSSH|PGP PRIVATE`).
- Novo check: `.env` **rastreado** falha (antes só staged).
- Calibrado empiricamente: zero falso-positivo no repo atual.

### 1.2 Higiene de commit no CI (modo mensagem)

`gate_commit_hygiene` ganhou modo CI: sem staged files e em `GITHUB_ACTIONS`/`--phase ci`, valida **mensagens de commit** (vocabulário AR-52 idêntico ao `validate-commit-msg.sh`):
- **HEAD = bloqueante** (trabalho novo);
- **Range `origin/<base>..HEAD` = advisory** — histórico pré-existente (ex.: commit de decom com "dev-claude") não trava o CI para sempre, e o squash-merge do closeout limpa o histórico no merge.
- Conteúdo do working tree NÃO é gateado em CI (o repo menciona vendors legitimamente em adapters/hooks).

### 1.3 Nova fase `ci` + workflow

`quality-gates.sh --phase ci` = gates de doc + segurança(árvore) + higiene(mensagens) — sem `gate_code` (o job `test` já roda a suíte; duplicar dobraria o tempo de CI). O job `governance` do `ci.yaml` (required check desde o ciclo 01) agora roda `--phase ci` com `fetch-depth: 0`. Resultado: **secret scan e higiene deixaram de ser bypassáveis com `--no-verify`**.

### 1.4 Estado dual dev-squad consolidado (por decisão, não por código novo)

Conforme a canonicity policy do `AGENT-MANIFEST.md` (markdown = canônico single-run; SQLite = canônico cross-run): os 5 tools do `mcp-hseos-swarm` são **filesystem-backed por design** (servem exatamente o escopo single-run), e o contrato de emissão dev-squad já é alcançável via MCP pelo `hseos-state-tracking` (`run_create`/`event_emit`). O wrapper morto `lib/run-state-dal.js` (zero importadores, confirmado por 2 mapeadores) foi **removido** e o README do server documenta a decisão. Fim da ambiguidade "integração SQLite pendente".
