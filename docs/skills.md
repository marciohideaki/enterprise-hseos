# Skills Catalog

> What gets enforced, when, and why. Skills are the governance layer that agents apply automatically based on what you're doing.

---

## How skills work

You don't activate skills directly — agents load them automatically when the task matches. The `SKILLS-REGISTRY.md` defines triggers for each skill.

**Two tiers:**
- **Tier 1 (SKILL-QUICK.md)** — Fast checklist, loaded by default. Low context cost.
- **Tier 2 (SKILL.md)** — Full algorithm with edge cases and remediation. Loaded for deep analysis or when fixing a violation.

If you see an agent reference a skill in its output (e.g., `[commit-hygiene] violation detected`), it has loaded the relevant tier and is applying the rules from that skill file.

## Skills by domain

### Code quality

#### `commit-hygiene`
**What it enforces:** Commit message format and content.
- Conventional commit format required (`feat:`, `fix:`, `chore:`, `docs:`, etc.)
- No AI tool names or co-authorship trailers (`Co-authored-by: Claude` → rejected)
- No HSEOS methodology terms in commit bodies
- Word-boundary matching — `"domain"` contains `"ai"` but is never flagged

**Triggers automatically when:** writing or amending a commit, generating PR title/body, generating changelog.

---

#### `sanitize-comments`
**What it enforces:** Source code comments must not contain methodology references or AI attribution.
- Removes: `FR-001`, `NFR-`, `Story:`, `Epic:`, `Sprint`, `// Generated by Claude`
- Zero changes to executable code — comments only

**Triggers automatically when:** cleaning comments before commit, preparing code for open-source or client delivery.

---

#### `test-coverage`
**What it enforces:** Test adequacy when new business logic is introduced.
- New use cases or domain rules require tests
- Test pyramid compliance (unit + integration balance)
- No coverage theater — validates real scenarios, not happy-path only

**Triggers automatically when:** PR adds new business logic, use cases, or domain rules.

---

#### `pr-review`
**What it enforces:** Pull request review standards.
- Risk triage before review depth allocation
- Quality gates (CI, coverage, lint)
- DDD boundary evidence
- Contract safety (breaking change detection)
- Blast radius analysis
- Adversarial analysis using the Socratic Questioning pattern (L1→L5) for high-risk PRs
- Governance compliance (ADR status if required)

**Triggers automatically when:** performing or generating a PR review.

---

### Architecture

#### `ddd-boundary-check`
**What it enforces:** DDD layer boundaries — no cross-layer violations.
- `domain/` must not import from `infrastructure/` or `api/`
- `application/` orchestrates; does not contain domain logic
- Bounded context isolation — no direct cross-module imports

**Triggers automatically when:** PR touches `domain/`, `application/`, `infrastructure/`; new modules; cross-module imports.

---

#### `breaking-change-detection`
**What it enforces:** Breaking changes in API and event contracts.
- Field removed or renamed
- Endpoint removed or path changed
- Enum value removed
- Event schema incompatibility

**Triggers automatically when:** PR modifies HTTP endpoints, event schemas, DTOs, shared contracts, gRPC definitions.

---

#### `adr-compliance`
**What it enforces:** ADR (Architecture Decision Record) format, completeness, and linkage.
- Required sections present (Context, Decision, Consequences)
- Approval status must be explicit
- Changes requiring an ADR must have one before proceeding

**Triggers automatically when:** creating, reviewing, or referencing an ADR; validating a change that requires one.

---

### Security

#### `secure-coding`
**What it enforces:** OWASP top-10 guardrails at the code level.
- No hardcoded secrets, tokens, or passwords
- New endpoints are not `AllowAnonymous` by default
- No SQL / NoSQL / command injection vectors
- No deprecated crypto algorithms (MD5, SHA1, DES, RC4)
- No PII in logs

**Triggers automatically when:** PR touches auth, tokens, secrets, PII, new endpoints, or new dependencies.

---

#### `threat-modeling`
**What it enforces:** Formal threat model — trust boundaries, attacker capabilities, abuse paths, mitigations.

> Explicit request only. Not auto-loaded for general security reviews.

**Triggers when:** user explicitly requests "threat model", "AppSec analysis", or "abuse path enumeration".

---

#### `dependency-audit`
**What it enforces:** New or upgraded dependencies.
- Known vulnerabilities (npm audit, nuget, etc.)
- License compatibility
- Version pinning

**Triggers automatically when:** PR adds or upgrades a third-party package, changes a lockfile.

---

#### `project-state`
**What it enforces:** Live project state and task backlog persistence across agent sessions.
- STATE.md and TASKS.md updated on every phase transition and agent hand-off
- Mode detection: hybrid auto-selects MCP → CLI → Markdown based on available tooling
- Pruning rule applied to TASKS.md when > 5 tasks completed
- Escalation logged when backend falls back

**Four modes:**
- `mcp-sqlite` — MCP server + SQLite (structured, atomic, best for teams)
- `cli-sqlite` — Shell/PowerShell + SQLite (no server, works in CI)
- `skill-only` — Direct markdown writes (zero infra)
- `hybrid` — Auto-detect with graceful fallback

**Triggers automatically when:** reading or writing STATE.md / TASKS.md, epic phase transitions, agent hand-offs.

### DevOps & delivery

#### `gitops-deploy`
**What it enforces:** GitOps manifest update discipline.
- Detect active deployment profile (centralized vs app-paired) before any action
- Branch naming must follow profile template
- Commit message must follow profile template
- Validation command must pass before commit
- PR base branch from `pr-base-map` in active profile

**Triggers automatically when:** deploy, image bump, tag update, promote version to any Kubernetes environment.

---

#### `gitops-add-service`
**What it enforces:** Adding a new service to an existing K8s project.
- Creates Deployment + Service + ConfigMap in base
- Updates all overlays (dev/hmg/stg/prod)
- Validates and commits in feature branch

**Triggers automatically when:** user wants to add a new service/worker/microservice to an existing K8s project.

---

#### `gitops-new-project`
**What it enforces:** Creating a complete GitOps structure for a new project.
- Namespaces, infra (StatefulSets), services (Deployments)
- Overlays for all environments
- ArgoCD Applications and AppProject
- Bootstrap script

**Triggers automatically when:** user wants to create a brand-new project in Kubernetes / GitOps.

---

#### `release-control`
**What it enforces:** Release governance — changelog discipline, risk classification, rollout requirements.
- CHANGELOG format and completeness
- SemVer discipline
- Hotfix procedures
- Rollback requirements for high-risk releases

**Triggers automatically when:** preparing a release, creating a release branch or tag, updating CHANGELOG.

### Process & documentation

#### `spec-driven`
**What it enforces:** 4-phase workflow for planning features — Specify → Design → Tasks → Implement.
- Work begins from specifications, not assumptions
- Each phase produces a typed artifact before the next starts

**Triggers automatically when:** starting a new feature, service, or significant change.

---

#### `documentation-completeness`
**What it enforces:** Documentation coverage for public code, APIs, and architecture.
- New public methods/classes require doc comments
- New endpoints require OpenAPI / Swagger entries
- New services require architecture documentation

**Triggers automatically when:** PR introduces new public code, new API endpoints, or new services.

---

#### `observability-compliance`
**What it enforces:** Structured logging, metrics, tracing, and alerting requirements.
- New endpoints/consumers require trace IDs and correlation IDs
- Logs must be structured (no string interpolation with raw objects)
- New background jobs require metrics

**Triggers automatically when:** PR adds new service operations, endpoints, background jobs, or event consumers.

---

#### `naming-conventions`
**What it enforces:** Code naming rules per stack.
- Stack-specific rules: C#, Dart, TypeScript, Java, Go, PHP, C++
- Semantic naming (names must reflect intent, not implementation detail)

**Triggers automatically when:** generating new code artifacts or reviewing naming in any stack.

---

#### `accessibility`
**What it enforces:** WCAG 2.1 compliance for Flutter and React Native UIs.
- Screen reader labels
- Minimum touch target sizes
- Color contrast ratios

**Triggers automatically when:** building or reviewing UI components in Flutter or React Native.

---

#### `performance-profiling`
**What it enforces:** Hot path performance standards.
- Zero-alloc validation patterns
- Lock-free pattern audit
- Benchmark regression gates

> Opt-in only — applies if the service has an active Performance Engineering ADR.

**Triggers automatically when:** PR touches hot path code, benchmarks, or lock-free data structures (in opted-in services).

### Governance

#### `agent-permissions`
**What it enforces:** Least-privilege `.claude/settings.json` generation.
- Read-only commands only by default
- Stack-aware (different defaults per project type)
- No absolute paths, no custom scripts in allowed commands

**Triggers automatically when:** setting up Claude Code for a new project, auditing existing settings.

---

#### `mcp-governance`
**What it enforces:** MCP tool call discipline across all agents.
- MCP first — always prefer MCP over equivalent CLI/curl
- Call budget per operation type (list ≤ 5, get ≤ 10, write ≤ 3)
- Loop guards — no unbounded iteration over MCP calls
- Rate limit handling — exponential backoff on 429

**Triggers automatically when:** multi-step workflow with MCP calls; deciding between MCP vs CLI.

---

#### `multi-agent-orchestration`
**What it enforces:** ORBIT pattern selection and workflow structure.
- Every workflow must declare prerequisites, outputs, and stop conditions
- 6 formal patterns: Sequential Chain, Parallel Fan-Out, Map-Reduce, Critic Loop, Routing, Human-in-the-Loop
- ORBIT never absorbs agent authority — it coordinates it

**Triggers automatically when:** ORBIT planning or executing a multi-agent workflow.

---

#### `inter-agent-comms`
**What it enforces:** Cross-agent communication protocol.
- Sequential hand-off via shared state files
- Real-time coordination via `claude-peers` MCP (when available)
- Sensitive artifacts must not be transmitted as inter-agent messages
- All cross-session communication logged in workflow state

**Triggers automatically when:** ORBIT coordinating multi-session hand-offs.

---

#### `policy-layer`
**What it enforces:** AI agent governance — spend caps, tool access, audit trail.
- Tool access matrix per agent role
- Spend cap and rate limit recommendations
- Destructive operations require Human-in-the-Loop gate
- Audit trail completeness check

**Owner:** SABLE  
**Triggers automatically when:** auditing agent governance; agent requests access outside its defined scope.

---

#### `context-policy`
**What it enforces:** Context budget governance per the AI-SDLC standard.
- Session context limits: Ideal ≤ 40%, Maximum ≤ 60%, Blocked > 60% (when middleware enforced)
- Per-agent sizing rules: GHOST (1 story = 1 session, max 200 lines), ORBIT (resume from state), BLITZ (scope must fit in one session)
- Stateless execution: every task must be resumable from its `input_contract` alone — no dependency on conversation history
- Session resume protocol: how to save state and re-enter a task cold from the contract

**Triggers automatically when:** session growing long; output quality degrading; sizing a task for a session; task contract is missing `input_contract`/`output_contract`.

---

#### `ai-observability`
**What it enforces:** AI usage observability and FinOps KPIs per the AI-SDLC standard.
- Native HSEOS metrics (no extra infra): gate failure rate, delivery cycle time, story completion rate
- Metrics requiring mission-control: `tokens_input`, `tokens_output`, `context_usage`, `tasks_executed`, `error_rate`
- KPIs: cost per feature, % stateless execution, context budget adherence, rework rate
- SABLE FinOps audit checklist at end of each epic

**Owner:** SABLE  
**Triggers automatically when:** auditing AI usage metrics; reviewing FinOps KPIs; SABLE running end-of-epic governance audit; configuring mission-control.

---

#### `second-brain`
**What it provides:** Optional integration with a personal knowledge vault. When enabled, HSEOS agents read strategic context from the vault and write decisions/learnings back to it in vault-compatible format.

**Read (by agents at task start):**
- ORBIT, NYX, BLITZ — `_memory/current-state.md` (live state bridge)
- VECTOR — `_knowledge/goals.md`, `_knowledge/about-me.md`
- CIPHER — all `_decisions/` ADRs + architecture learnings
- GHOST — domain-relevant `_learnings/` files
- RAZOR — `_knowledge/projects.md`

**Write (ORBIT, CIPHER, QUILL only — after human approval):**
- Architecture decisions → `_decisions/hseos/YYYY-MM-DD-{name}.md`
- Epic learnings → `_learnings/hseos-{topic}.md`
- Epic summary → appended to `_memory/current-state.md` via `hseos brain sync`

**Zero-impact default:** `enabled: false` — no vault, no behavior change.  
**Fallback:** If vault unavailable → skip silently, continue with HSEOS sources only.

**CLI:** `hseos brain status` | `hseos brain sync`

**Triggers automatically when:** `second_brain.enabled = true` + agent starting task; CIPHER or QUILL completing epic work.

## Skill violation → what happens?

When a skill check fails, the agent:

1. **Stops** — does not proceed to the next step
2. **Reports** — exact file and line if applicable, rule violated, severity
3. **Guides** — what needs to change before proceeding

Agents never auto-fix governance violations silently. You will always see what was blocked and why.
