> Common issues, error messages, and answers to frequent questions about HSEOS v2.0.0.

## Installation

### `npx hseos install` fails with "command not found"

**Cause:** Node.js not installed or version < 20.

```bash
node --version   # must be â‰Ą 20.0.0
```

Install Node.js via [nvm](https://github.com/nvm-sh/nvm):

```bash
nvm install 20
nvm use 20
```

---

### Install completes but `.enterprise/` is missing

**Cause:** The install ran from outside the project root, or a previous partial install left a corrupt state.

```bash
# Verify you are in the project root
ls package.json   # should exist

# Re-run install (idempotent â€” safe to run multiple times)
npx hseos install

# Validate after install
npx hseos validate
```

---

### `hseos validate` shows "46 skills registered" but agent commands are missing

**Cause:** The adapter compilation step failed silently (often a missing `@hseos/adapter-*` package or a schema mismatch).

```bash
# Check the compiler output
npx hseos agent-core compile --verbose

# Check which adapters are registered
npx hseos agent-core list-adapters
```

## Quality Gates

### Commit rejected: "Direct commits to main/master are not allowed"

This is expected behaviour. HSEOS enforces branch protection via the pre-commit hook.

```bash
git checkout -b feature/my-task
git commit -m "feat(scope): description"
```

---

### Commit rejected: "AI tool attribution detected in commit message"

HSEOS blocks commit messages that contain `Co-authored-by: Claude`, `Co-authored-by: Codex`, `Generated by`, etc.

Remove the attribution from your commit message. If you need to credit a human collaborator:

```bash
git commit -m "feat(scope): description" --trailer "Co-authored-by: Name <email>"
```

---

### Commit rejected: "Commit message format invalid"

Required format: `<type>(<scope>): <summary>`

Valid types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `ci`, `build`

```bash
# Valid examples
git commit -m "feat(auth): add JWT refresh endpoint"
git commit -m "fix(state): prevent orphan run on crash"
git commit -m "chore(deps): upgrade better-sqlite3 to 12.9.0"
```

---

### Pre-commit runs `eslint` but the command is not found

```bash
npm install   # ensure devDependencies are installed
```

If you are in a CI container (e.g., `node:20`), `eslint` is a devDependency â€” it is not globally available. Either run `npm ci` before committing, or install it globally:

```bash
npm install -g eslint
```

---

### Smoke test fails with "jq: command not found"

The standalone smoke CI runs in a `node:20` container where `jq` is not pre-installed. The `.github/workflows/standalone-smoke.yaml` has a step to install it:

```yaml
- name: Install system dependencies
  run: apt-get update -qq && apt-get install -y -qq jq
```

If you are running tests locally in a minimal environment:

```bash
# Ubuntu / Debian
sudo apt-get install -y jq

# macOS
brew install jq
```

## State Tracking

### `hseos kanban` shows no runs

**Cause:** No runs have been created yet, or the database path is wrong.

```bash
# Check where the DB is expected
echo $HSEOS_STATE_DB   # should be set, or defaults to .hseos/state/project.db

# Check if the DB file exists
ls .hseos/state/project.db

# Check what runs exist
hseos state list --status all
```

---

### The web kanban at `localhost:3200` shows stale data

The UI updates via SSE (Server-Sent Events). If the feed appears stale:

1. Hard-refresh the browser (`Ctrl+Shift+R`)
2. Check that the `state-ui` process is still running (`ps aux | grep state-ui`)
3. If the state-ui process crashed, restart it: `hseos state-ui`

---

### `state-emit-hook.sh` logs "better-sqlite3: Cannot find module"

**Cause:** `better-sqlite3` is not installed. It is a required `dependency` (not `optionalDependency`) since v2.0.0.

```bash
npm install   # installs better-sqlite3 from package.json dependencies
```

If you are in a clean CI environment:

```bash
npm ci
```

---

### Session context is not auto-detected at SessionStart

The hook requires:
1. `better-sqlite3` installed (see above)
2. An active run in `as_runs` (`status = 'active'`)
3. The DB path is accessible (default: `.hseos/state/project.db`)

To debug:

```bash
# Manually test the hook
CLAUDE_HOOK_EVENT=SessionStart bash scripts/governance/state-emit-hook.sh

# Check active runs
hseos state list --status active
```

## MCP Servers

### MCP server not discovered by Claude Code

Check `.mcp.json` at the project root:

```bash
cat .mcp.json   # should list hseos-governance and/or hseos-swarm
```

If missing, add the servers manually:

```json
{
  "mcpServers": {
    "hseos-governance": {
      "command": "node",
      "args": ["tools/mcp/hseos-governance/server.js"]
    },
    "hseos-swarm": {
      "command": "node",
      "args": ["tools/mcp/hseos-swarm/server.js"]
    }
  }
}
```

Then restart Claude Code to reload MCP connections.

---

### MCP tool returns "HSEOS_STATE_DB not set"

Set the env variable in `.mcp.json`:

```json
{
  "mcpServers": {
    "hseos-swarm": {
      "command": "node",
      "args": ["tools/mcp/hseos-swarm/server.js"],
      "env": {
        "HSEOS_STATE_DB": ".hseos/state/project.db"
      }
    }
  }
}
```

## Self-Verification

### `hseos verify` reports "git hooks not installed"

```bash
# Re-install hooks (idempotent)
npx hseos install

# Or directly
npm run prepare   # runs husky install
```

---

### `hseos doctor` reports "axon index absent" â€” should I worry?

No. The `hseos-axon-bridge` MCP server requires an Axon knowledge index (`.axon/`). If your project does not use Axon, this warning is expected and harmless.

To suppress it, set in `hseos.config.yaml`:

```yaml
mcp:
  axon_bridge:
    enabled: false
```

## Adapter SDK

### My custom adapter is not discovered by the compiler

The compiler discovers adapters matching the prefix `@hseos/adapter-*` in `node_modules/`. Verify:

```bash
ls node_modules/@hseos/   # should include your adapter package

# Check what adapters the compiler found
npx hseos agent-core list-adapters
```

If your adapter is a local package (not published to npm), link it first:

```bash
cd packages/adapter-myadapter && npm link
cd /path/to/project && npm link @hseos/adapter-myadapter
```

---

### Round-trip test fails with "zero diff not satisfied"

Your `emit()` method is not idempotent â€” running it twice produces a different output. Common causes:

- Writing timestamps or random values to generated files
- Not sorting arrays before writing (order varies between runs)
- Missing a clean step before writing (appending instead of overwriting)

Use `AdapterBase.writeFile(path, content)` instead of raw `fs.writeFile` â€” it handles idempotency checks.

## Plugins

### `hseos plugin doctor` reports conformance errors

Plugin conformance checks validate the plugin manifest against `.claude-plugin/` and `.codex-plugin/` schemas.

```bash

# Re-install a specific plugin
hseos plugin install <plugin-id>
```

## Frequently Asked Questions

### Can I use HSEOS without Claude Code?

Yes. The governance layer (`.enterprise/`) and quality gates (git hooks) work with any editor. The agent commands (`.claude/commands/`) are Claude Code-specific, but the agents themselves are documented in `docs/agents/` and can be activated manually in any context where you have an AI assistant.

Use `npx hseos install --tools codex` (or `cursor`, `windsurf`, etc.) to generate adapters for your preferred tool.

---

### Do I need to activate agents manually?

No. Agents are not daemons â€” they are behavioural contracts. When you open a conversation with an AI assistant and mention a task that matches an agent's domain, the agent's skills load automatically from the registry. You can also activate explicitly by typing the agent's slash command (e.g., `/ghost` for implementation tasks).

---

### Can multiple agents run simultaneously?

Yes, via `SWARM`. SWARM decomposes a heterogeneous batch of independent tasks and dispatches one isolated Sonnet subagent per task into a separate `.worktrees/<task-id>/` directory. Tasks run in parallel without filesystem collisions. See `docs/agents/swarm.md` for the protocol.

---

### Is NPM publish supported?

Not by default. This is an institutional framework installed from source or via git dependencies. The `release.yaml` CI workflow has the NPM publish step present but requires a `NPM_TOKEN` secret â€” deliberately not configured for private/institutional use.

---

### Where are agent decisions recorded?

In `.enterprise/.specs/decisions/` as ADR files. Every architectural trade-off, governance exception, or breaking change requires an ADR before implementation. Agents cannot take those decisions autonomously â€” they produce an ADR draft and stop, waiting for human approval.

---

### How do I add a new skill?

1. Use the `hseos-skill-creator` plugin: `hseos plugin install hseos-skill-creator`, then `/skill-new` in Claude Code
2. The plugin scaffolds `SKILL.md` + `SKILL-QUICK.md` with correct HSEOS frontmatter
3. Register the skill in `.enterprise/governance/agent-skills/SKILLS-REGISTRY.md`
4. Run `npm run validate:schemas` to verify

---

### How do I override an agent's authority?

Agent authority is defined in `.enterprise/agents/<code>/authority.md`. To request an exception:

1. Open `.enterprise/.specs/decisions/` and create an ADR from `_TEMPLATE.md`
2. Document the rationale and the specific override
3. Get human approval (PR review)
4. After approval, update the agent's authority file accordingly

Agents cannot self-modify their authority â€” stop conditions apply.
