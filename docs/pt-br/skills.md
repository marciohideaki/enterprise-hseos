# Catálogo de Skills

> O que é aplicado, quando e por quê. Skills são a camada de governança que os agentes aplicam automaticamente com base no que você está fazendo.

---

## Como as skills funcionam

Você não ativa skills diretamente — os agentes as carregam automaticamente quando a tarefa corresponde. O `SKILLS-REGISTRY.md` define os gatilhos para cada skill.

**Dois níveis:**
- **Nível 1 (SKILL-QUICK.md)** — Checklist rápido, carregado por padrão. Baixo custo de contexto.
- **Nível 2 (SKILL.md)** — Algoritmo completo com casos extremos e remediação. Carregado para análise profunda ou ao corrigir uma violação.

Se você ver um agente referenciar uma skill em seu output (ex: `[commit-hygiene] violation detected`), ele carregou o nível relevante e está aplicando as regras daquele arquivo de skill.

## Skills por domínio

### Qualidade de código

#### `commit-hygiene`
**O que aplica:** Formato e conteúdo de mensagens de commit.
- Formato de conventional commit obrigatório (`feat:`, `fix:`, `chore:`, `docs:`, etc.)
- Sem nomes de ferramentas de IA ou trailers de co-autoria (`Co-authored-by: Claude` → rejeitado)
- Sem termos de metodologia HSEOS nos corpos de commit
- Correspondência por limite de palavra — `"domain"` contém `"ai"` mas nunca é sinalizado

**Aciona automaticamente quando:** escrevendo ou alterando um commit, gerando título/corpo de PR, gerando changelog.

---

#### `sanitize-comments`
**O que aplica:** Comentários no código-fonte não devem conter referências de metodologia ou atribuição de IA.
- Remove: `FR-001`, `NFR-`, `Story:`, `Epic:`, `Sprint`, `// Generated by Claude`
- Zero alterações no código executável — apenas comentários

**Aciona automaticamente quando:** limpando comentários antes do commit, preparando código para open-source ou entrega ao cliente.

---

#### `test-coverage`
**O que aplica:** Adequação de testes quando nova lógica de negócio é introduzida.
- Novos casos de uso ou regras de domínio exigem testes
- Conformidade com pirâmide de testes (equilíbrio entre unitário e integração)
- Sem cobertura teatral — valida cenários reais, não apenas o caminho feliz

**Aciona automaticamente quando:** PR adiciona nova lógica de negócio, casos de uso ou regras de domínio.

---

#### `pr-review`
**O que aplica:** Padrões de revisão de pull request.
- Triagem de risco antes de alocar profundidade de revisão
- Quality gates (CI, cobertura, lint)
- Evidência de fronteira DDD
- Segurança de contrato (detecção de breaking change)
- Análise de blast radius
- Análise adversarial usando o padrão de Questionamento Socrático (L1→L5) para PRs de alto risco
- Conformidade de governança (status de ADR se necessário)

**Aciona automaticamente quando:** realizando ou gerando uma revisão de PR.

---

### Arquitetura

#### `ddd-boundary-check`
**O que aplica:** Fronteiras de camada DDD — sem violações entre camadas.
- `domain/` não deve importar de `infrastructure/` ou `api/`
- `application/` orquestra; não contém lógica de domínio
- Isolamento de bounded context — sem imports diretos entre módulos

**Aciona automaticamente quando:** PR toca `domain/`, `application/`, `infrastructure/`; novos módulos; imports entre módulos.

---

#### `breaking-change-detection`
**O que aplica:** Breaking changes em contratos de API e eventos.
- Remoção de campos — detectada em schemas de request/response
- Renomeação de endpoints — verificada contra mapa de rotas publicadas
- Mudanças de payload de evento — verificadas contra consumers conhecidos

**Aciona automaticamente quando:** PR modifica controllers, DTOs, schemas de evento ou interfaces de contrato.

---

### Documentação

#### `doc-project`
**O que aplica:** Geração de documentação rica e profissional para projetos.
- README bilíngue EN+PT-BR
- Docs estruturados com placeholders de imagem com dimensões corretas
- CHANGELOG, CONTRIBUTING, SECURITY e templates GitHub

**Aciona automaticamente quando:** criando documentação de projeto, gerando README, atualizando docs existentes.

---

#### `repo-radar`
**O que aplica:** Análise e classificação de repositório GitHub.
- Avalia maturidade, qualidade e adotabilidade do repositório
- Registra veredito em PROJECT_EVALUATIONS.md

**Aciona automaticamente quando:** avaliando um repositório externo para adoção ou referência.

---

#### `tech-research`
**O que aplica:** Pesquisa e avaliação de tecnologias.
- Comparação de alternativas com trade-offs explícitos
- Recomendação fundamentada com critérios de decisão

**Aciona automaticamente quando:** pesquisando tecnologias, ferramentas ou abordagens técnicas.

---

#### `rfc`
**O que aplica:** Geração de RFC (Request for Comments) ou design docs.
- Estrutura: problema, proposta, alternativas, trade-offs, impacto, métricas de sucesso

**Aciona automaticamente quando:** documentando decisões técnicas ou arquiteturais.

---

### Segurança

#### `agent-permissions`
**O que aplica:** Controle de permissões de agentes.
- Paths absolutos proibidos em definições de skill
- Escopo mínimo de permissões por agente
- Isolamento de worktree para execução paralela

**Aciona automaticamente quando:** definindo ou auditando permissões de agentes.
