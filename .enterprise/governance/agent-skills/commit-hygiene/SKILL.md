## When to use

## 1. Hard Rules

### 1.1 No Co-authorship Trailers

NEVER include any trailer that credits a tool, assistant, or AI system:

```
# Forbidden:
Co-authored-by: Claude <noreply@anthropic.com>
Co-authored-by: GitHub Copilot <copilot@github.com>
Co-authors: [any name]
Assisted-by: [any name]
Generated-by: [any name]
```

This applies even when the AI contribution was significant. The commit represents the
engineering team's decision to ship the change â€” not the tool used to draft it.

### 1.2 No AI Tool Mentions

Never mention AI tools, AI models, or AI frameworks in commit subjects, bodies,
PR titles, PR descriptions, or changelogs â€” **unless the commit is literally implementing
integration with that tool** (e.g., adding an OpenAI SDK integration to the product itself).

## 2. Matching Algorithm â€” Word-Boundary Rule (Critical)

**All term checks MUST use whole-word matching (word boundary: `\b`).**

### 2.1 Why This Matters

Substring matching produces false positives on legitimate technical vocabulary:

| Legitimate word | Contains | Substring match (WRONG) | Word-boundary match (CORRECT) |
|---|---|---|---|
| `domain` | `ai` | âŒ FAIL | âœ… PASS |
| `email` | `ai` | âŒ FAIL | âœ… PASS |
| `detail` | `ai` | âŒ FAIL | âœ… PASS |

| `cursor` (DB) | `cursor` | âŒ FAIL | âœ… PASS |
| `agent` (pool) | `agent` | âŒ FAIL | âœ… PASS |

Flagging `domain` because it contains `ai` is a **hard bug**. Never implement
AI-mention detection as a case-insensitive `contains("ai")` check.

## 4. Commit Message Format

```
<type>(<scope>): <imperative summary>

[optional body: what changed and why]

[optional footer: ticket refs, e.g., Closes #123]
```

**Types:** `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `perf`, `ci`, `build`, `revert`

**Rules:**
- Subject: max 72 characters
- Subject: imperative mood ("add", not "added" or "adds")
- Body: explains the *why*, not the *how*
- Body: does NOT describe the tool or process used
- No trailing period on subject line


## 6. Remediation

If a forbidden term is present:

1. **Identify** the specific term that triggered the rule
2. **Verify** it is a true violation (whole-word match, not substring)
3. **Rewrite** the message: remove AI/tool attribution, preserve only the technical change summary
4. **Do not** add any explanation that reveals the original message contained AI attribution

### Examples

```

feat(auth): implement JWT validation as suggested by Claude

# PASS â€” fixed
feat(auth): implement JWT validation using HS256 with 15-minute expiry

# FAIL â€” co-authorship trailer

# PASS â€” fixed (trailer removed, message unchanged)

# FALSE POSITIVE â€” "domain" contains "ai" but is NOT a violation

#           ^^^^^^ PASS â€” "domain" is a legitimate technical term
```
